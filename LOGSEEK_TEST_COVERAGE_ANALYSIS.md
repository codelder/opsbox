# LogSeek 测试覆盖分析报告

## 执行摘要

**结论**：当前测试覆盖**不足以支持安全重构**。虽然有大量单元测试，但在**查询语法组合**、**多数据源交叉**和**边界条件**方面存在明显缺口。

---

## 一、测试分层现状

### 1.1 单元测试 (Rust `#[test]`)

#### ✅ **覆盖良好的模块**

| 模块 | 测试数量 | 覆盖内容 |
|------|---------|---------|
| **query/parser.rs** | 30+ | 查询语法解析、优先级、路径过滤、错误处理 |
| **query/lexer.rs** | 15+ | 词法分析、Token 识别、边界情况 |
| **service/nl2q.rs** | 20+ | 自然语言转查询 |
| **api/models.rs** | 13+ | 数据模型序列化/反序列化 |
| **domain/config.rs** | 6 | 配置解析 |

#### ⚠️ **覆盖不足的模块**

| 模块 | 问题 |
|------|------|
| **service/search.rs** | 只有少量测试，核心搜索逻辑未充分覆盖 |
| **service/search_executor.rs** | 执行器的并发、取消、错误恢复未测试 |
| **service/entry_stream.rs** | 流处理的边界条件（空文件、超大文件）未测试 |
| **agent/mod.rs** | Agent 通信的网络故障、超时场景未覆盖 |

---

### 1.2 集成测试 (E2E with Real Backend)

#### ✅ **已覆盖的场景**

| 测试文件 | 数据源 | 查询类型 | 文件类型 |
|---------|--------|---------|---------|
| `integration_local.spec.ts` | Local | `app:` directive | `.log`, `.tar`, `.gz` |
| `integration_agent.spec.ts` | Agent | `app:` directive | `.log`, `.tar`, `.tar.gz`, 多 `.gz` |
| `s3_archive.spec.ts` | S3 | `app:` directive | `.tar.gz` 归档 |
| `integration_mixed.spec.ts` | Local + Agent + S3 | `app:` directive | 混合源 |
| `integration_relative_glob.spec.ts` | Local | `app:` + 相对路径 glob | - |

#### ❌ **未覆盖的关键场景**

1. **查询语法组合**
   - ❌ 布尔运算符组合：`(foo OR bar) AND -baz`
   - ❌ 正则表达式：`/ERR\d+/`
   - ❌ 短语搜索：`"exact phrase"`
   - ❌ 路径过滤：`path:logs/**/*.log`
   - ❌ 复杂嵌套：`((a OR b) AND (c OR (d AND -e))) OR -(f OR g)`

2. **多源查询组合**
   - ❌ 同时搜索 Local + Agent + S3 并验证结果合并
   - ❌ 不同源返回速度差异时的 UI 更新
   - ❌ 部分源失败时的降级处理

3. **边界条件**
   - ❌ 空结果集
   - ❌ 超大结果集（>50,000 行）
   - ❌ 超长单行（>1MB）
   - ❌ 二进制文件误识别
   - ❌ 损坏的归档文件

4. **并发与取消**
   - ❌ 快速连续发起多次搜索
   - ❌ 搜索进行中切换查询
   - ❌ 浏览器关闭标签页时的资源清理

### 1.3 UI 交互测试 (Mocked E2E)

## 三、关键缺失场景详解

### 3.1 查询语法在真实搜索中的验证

**当前状态**：
```typescript
// ❌ 不存在这样的测试

  await searchInput.fill('(ERROR OR WARN) AND -deprecated path:logs/**/*.log');
  await searchInput.press('Enter');

  // 验证结果只包含 logs/ 下的文件
  // 验证每个结果都包含 ERROR 或 WARN
  // 验证没有结果包含 deprecated
```

**影响**：无法确保查询解析器和搜索执行器的集成正确性。

## 四、重构风险评估

### 4.1 高风险区域（测试覆盖不足）

| 模块 | 风险等级 | 原因 |
|------|---------|------|
| **SearchExecutor** | 🔴 高 | 并发、取消、错误处理逻辑复杂，缺少集成测试 |
| **查询语法集成** | 🔴 高 | 单元测试充分，但缺少端到端验证 |
| **多源协调** | 🟡 中 | 有基本测试，但缺少复杂场景 |
| **归档处理** | 🟡 中 | 部分覆盖，但边界条件未测试 |

### 4.2 低风险区域（测试覆盖充分）

| 模块 | 风险等级 | 原因 |
|------|---------|------|
| **查询解析器** | 🟢 低 | 30+ 单元测试，覆盖全面 |
| **词法分析器** | 🟢 低 | 15+ 单元测试，边界条件完善 |
| **数据模型** | 🟢 低 | 序列化/反序列化充分测试 |

## 七、结论与行动建议

### 当前状态
- ✅ **查询解析器**：单元测试充分，可安全重构
- ⚠️ **搜索执行器**：缺少集成测试，重构有风险
- ❌ **查询语法集成**：无端到端验证，**不建议重构**
- ❌ **多源协调**：复杂场景未覆盖，**不建议重构**

### 行动建议
1. **立即补充 P0 测试**（预计 2-3 天）
2. **重构前运行完整测试套件**
3. **重构后对比测试结果**，确保无回退
4. **逐步补充 P1/P2 测试**，提升长期可维护性

### 重构策略
- 优先重构**查询解析器**（测试充分）
- 延后重构**搜索执行器**（补充测试后）
- **禁止**重构查询语法集成逻辑（测试缺失严重）
